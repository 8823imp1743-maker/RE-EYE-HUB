# RE-EYE-HUB プロダクト憲法（Cursor 用・完成版 v1.1）

> 事業方針と設計方針を分離し、将来のブレ（スニーカー専用化・検索エンジン化・通知乱発・広告まみれ化）を防ぐ。

---

# 最上位原則

RE-EYE-HUB の目的は **「通知を送ること」ではない**。

**「ユーザーの購入成功率を最大化すること」** である。

すべての機能追加・UI変更・通知設計・収益化判断はこの目的に従う。

- 購入成功率を下げる機能は実装しない。
- 購入成功率を上げる機能を優先する。

復活通知・見張り台・オークション相場確認・サイズゲート・attribute-gate・通知高速化は、すべてこの目的のための手段である。

---

# 事業方針（プロダクト憲法）

## 本質

RE-EYE-HUB はスニーカー監視アプリでも、26.5cm監視アプリでもない。

**「欲しい条件の商品が復活した瞬間を知らせる監視プラットフォーム」**

ユーザーが欲しいのは：

- 復活通知
- 購入成功
- 安心感

検索結果そのものではない。

## キャッチコピー候補

- 欲しい条件だけ、復活した瞬間に知らせる。
- 毎日探さなくていい。
- 売り切れとの戦いを自動化する。
- あなたの代わりに24時間見張り続ける。

## ユーザーの理想体験

**ユーザーがやること：** 商品名入力 → 条件選択 → 監視開始

**システムがやること：** 24時間監視 → 復活検知 → 属性一致確認 → 通知

## 優先順位（この順序を崩さない）

1. 誤通知ゼロ
2. 属性一致精度
3. 復活検知速度
4. 購入成功率
5. 監視対象モール拡大（楽天 / Yahoo / Amazon / 公式 / ABC-MART / atmos / BILLY'S 等）
6. UI改善
7. 収益化（最後）

## 長期ポジション

在庫検索ツールではない。

**「欲しい条件の商品をユーザーの代わりに24時間見張り続ける復活監視プラットフォーム」**

---

# 設計方針（実装憲法）

## コアモデル

監視対象 = **商品名 + 属性 + 状態変化**

（旧来の「商品名 + サイズ」だけで考えない）

例：

| カテゴリ | 商品 | 属性 | 状態変化 |
|----------|------|------|----------|
| スニーカー | Air Max 90 | 26.5cm | 在庫なし → 在庫あり |
| アパレル | Tシャツ | XL | 売切れ → 復活 |
| フィギュア | ドラゴンボール限定版 | 初回特典版 | 予約終了 → 予約再開 |
| 家電 | PS6 | 本体 | 売切れ → 復活 |

## データモデル移行

| 現在 | 将来 |
|------|------|
| `shoeSize` 中心 | `targetAttributes` 中心 |

```js
targetAttributes = {
  size: "26.5",
  color: "black",
  model: "IM3110-500"
}
```

将来属性：`size` / `color` / `model` / `edition` / `version` / `capacity` / `specification` / `keyword`

## ゲート設計

| 現在 | 将来 |
|------|------|
| `shoe-size-gate` | `attribute-gate` |

`attribute-gate` の役割：通知前に「商品属性」と「監視条件」を比較。不一致 → `attribute_gate_skip`（通知しない）

---

# 属性一致原則

**在庫あり判定よりも属性一致判定を優先する。**

誤通知1件は正通知10件の価値を破壊する。

そのため：

- `attribute_mismatch` と判定された商品は **在庫があっても通知しない**。
- 対象属性に確信が持てない場合は **通知しない方を選ぶ**。

---

# 見張り台原則

**検索結果0件は失敗ではない。**

条件に一致する商品が現在存在しない場合は `watch_tower` として監視候補を提示する。

RE-EYE-HUB は検索ツールではなく監視ツールであるため、

- 「今買えない」より
- **「復活したら知らせる」**

を優先して体験設計する。

---

# 検索の役割

検索は本体ではない。**監視登録の入口**。

検索結果を見せるためのアプリではない。購入成功率を上げるためのアプリである。

---

# Phase 1 — 見張り台UI（検索結果 tier）

| tier | 意味 | 表示 |
|------|------|------|
| `buy_now` | 今すぐ購入可能 | 表示 |
| `watch_tower` | 条件一致・現在OOS・監視推奨 | 表示（主役） |
| `check_stock` | 在庫判定が曖昧 | 表示 |
| `inquiry` | 取り寄せ・問い合わせ | 表示 |
| `attribute_mismatch` | サイズ/色/型番違い | **非表示** |

## 技術分担（固定）

- **手動検索：** モール一覧のみ（PDPスキップ、Vercel 10秒制約）
- **Cron監視：** 厳密PDP + attribute-gate + 通知

---

# Phase 2 — attribute-gate（通知エンジン）

検索 → 監視登録 → Cron → 通知の全ルートで `targetAttributes`（model / color / size）を保持。

通知直前に `evaluateAttributeGate()` を実行。3軸いずれか不一致 → `attribute_gate_skip` ログを出して通知しない。

実装：`functions/lib/attribute-gate.js`（`monitor.js` / `poll.js`）

---

# 実装判断チェックリスト

変更を入れる前に必ず問う：

1. これは購入成功率を上げるか？
2. 属性不一致の誤通知リスクは増えないか？
3. 検索を主役にしていないか（監視入口のままか）？
4. 0件・OOS を「失敗」として扱っていないか？
5. 収益化が精度・体験より優先されていないか？

---

# ブレ防止リスト（やらないこと）

| やらないこと | 理由 |
|--------------|------|
| スニーカー / 26.5cm 専用化 | 汎用監視プラットフォームが本質 |
| 検索エンジン化（結果表示の肥大化） | 入口に徹する |
| 通知乱発（在庫曖昧・属性不明でも送る） | 誤通知が購入成功率を破壊 |
| 広告まみれ化（有料体験の毀損） | 収益化は最後 |
